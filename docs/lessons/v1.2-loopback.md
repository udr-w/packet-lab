## Session narrative

### Teaching-order mistake and correction

The session opened with a jargon mistake: the student was asked to predict
behavior on `lo` before `lo` had been defined. The student pushed back ("I do not
understand the question"). Corrected by first explaining `lo` — the loopback
interface, a virtual interface with no physical hardware, used for `127.0.0.0/8`
traffic — before re-asking the prediction.

**Student preference recorded (now in AGENTS.md under Mentor Behaviour):** the
student prefers this theory-before-prediction, jargon-defined-first style —
described it as "easier, closer and practical."

### The wlp0s20f3-vs-lo prediction

With `lo` defined, the student correctly predicted, with correct reasoning:
capturing on `wlp0s20f3` while pinging `127.0.0.1` should show nothing, because
loopback traffic never reaches the network wire.

A capture screenshot on `wlp0s20f3` (6 packets, 3 echo requests, 3 echo replies)
was shown — but on checking with the student, this was confirmed to be from
pinging the **router**, not `127.0.0.1`. It does not test the loopback prediction
and must not be treated as evidence either way. **This means the direct
`wlp0s20f3` + `127.0.0.1` test has still never actually been run.**

### The 0-packets result and the hidden-stderr bug (BUG 1 — FIXED)

The student then ran `sudo python3 scripts/packetlab.py lo` with
`ping -c 3 127.0.0.1` and saw **no packets** — and admitted the earlier "it
worked" report had been an unverified assumption, not an actual observation.

**Root cause (a tooling bug, not a networking fact):** `packetlab/capture.py` ran
tcpdump with `stderr=None` (inherited from the terminal), but
`scripts/packetlab.py` immediately opens a Rich `Live` display with `screen=True`,
which switches to the terminal's alternate screen buffer. Any startup message or
fatal error tcpdump printed to stderr showed for a fraction of a second and was
then covered by the alternate screen — so a failed capture looked *identical* to
"no traffic captured." This affected **every** interface, not just `lo`; v1.1 had
simply gotten lucky that `wlp0s20f3` never failed to start.

**Fix:**

- `packetlab/capture.py` now pipes tcpdump's stderr instead of losing it.
- `scripts/packetlab.py` now reports tcpdump's exit code and any stderr output
  once the capture ends (unless the student stopped it with Ctrl+C).

**Standing student directive recorded (AGENTS.md, "Division of Responsibility"):**
the student will never read, edit, or debug repository files — that is entirely
the assistant's responsibility, including preparing/fixing tooling before asking
the student to run anything. The student's role is limited to running commands,
observing, and answering conceptual questions.

### Re-run confirming the fix — 12 packets (DONE)

Re-run after the stderr fix: two `ping -c 3 127.0.0.1` runs on `lo` produced
**12 total packets, 6 echo requests, 6 echo replies, 0 unparsed lines.** This
confirmed both the stderr fix and the student's original loopback prediction.

### The my_ips / 127.0.0.1 counting bug (BUG 2 — FIXED, not re-verified live)

The same 12-packet result exposed a second bug: "Sent by me" and "Received by me"
both showed **0** despite 12 real packets.

**Cause:** `HostResolver._get_my_ips()` builds its set from `hostname -I`, which
never includes `127.0.0.1` (it only lists real NIC addresses). Since loopback ICMP
has source == destination == `127.0.0.1`, and that address was not in `my_ips` at
all, neither branch of `PacketStats.observe()`'s `if/elif` matched.

**Fix:** `resolver.py` now always adds `127.0.0.1` to `my_ips`. **Not yet
re-verified against a live capture.**

### The "everything counts as sent" nuance (OPEN — to raise with student)

A follow-on consequence, not yet discussed with the student: because loopback
source and destination are identical, and `observe()` uses
`if source in my_ips: sent += 1; elif destination in my_ips: received += 1`,
after the fix **every** loopback packet — both requests and replies — lands in
`sent`, and `received` stays 0.

This is **not** a bug to silently patch further; it is a genuine consequence of
talking to yourself, and is a good conceptual question for the sent/received
milestone: what do "sent" and "received" even mean when source and destination
are the same machine?

### Passwordless-sudo constraint and the setcap resolution (IN PROGRESS)

**Constraint discovered:** the assistant does not have passwordless sudo on this
machine, so it cannot run `sudo tcpdump` / the packetlab viewer itself to
independently verify live captures. The assistant can read, edit, and statically
verify code (`python3 -m compileall`, sample-line parsing), but the student must
run anything needing root and report results back. This was a large source of
friction this session.

The student asked how to give the assistant real capture ability without breaking
safety boundaries. **Agreed approach:** grant the `tcpdump` binary the
`cap_net_raw,cap_net_admin` Linux capabilities via `setcap` (the same technique
Wireshark's `dumpcap` uses for non-root capture), instead of sudoers/NOPASSWD or
sharing a password. One-time command, must be run by the student (needs their
sudo password interactively):

```bash
sudo setcap cap_net_raw,cap_net_admin+eip /usr/bin/tcpdump
```

This scopes elevated capability to that single binary only — no standing root
access, no sudoers change. **Caveat:** `apt upgrade` replacing the tcpdump binary
resets this; check with `getcap /usr/bin/tcpdump` and re-apply if empty.

To match, `scripts/packetlab.py` no longer hard-requires `os.geteuid() == 0`
(removed the check and the `os` import). It now just starts tcpdump and reports
failure via the stderr-surfacing added earlier this session if permissions are
missing. This also incidentally removes the old `ModuleNotFoundError: rich` under
`sudo` problem, since running without `sudo` avoids the root/user Python
environment split.

**As of end of session, the student had NOT yet confirmed running the `setcap`
command.** Confirm this first next session — once done, the assistant should be
able to run captures directly (no sudo) for independent verification.

## Session 2026-07-04c — v1.2 CLOSED

All STILL OPEN items resolved this session:

- **setcap confirmed applied** (`getcap` shows `cap_net_admin,cap_net_raw=eip`).
  The assistant verified non-root capture live and now runs captures itself for
  independent verification. sudo is no longer needed by anyone for the viewer.
- **BUG 3 found and fixed — invisible packet table.** The student reported the
  viewer "only shows a stat". Cause: the stats panel (~15 lines) rendered ABOVE
  the packet table inside Rich's alternate screen, which crops overflow — on a
  normal-height terminal the table was cut off entirely; and on Ctrl+C the
  alternate screen vanished with everything on it. Fix in `packetlab/ui.py` +
  `scripts/packetlab.py`: evidence-first ordering (table above stats),
  height-adaptive row budget (`row_budget()`), and the final screen is reprinted
  to the normal buffer after Ctrl+C so evidence persists. Verified live.
- **The real `wlp0s20f3` + `127.0.0.1` test finally ran** (student, then
  independently reproduced by the assistant with a `host 127.0.0.1` filter):
  30s capture, **0 packets** while `ping -c 3 127.0.0.1` completed normally.
  Prediction confirmed with direct evidence for the first time.
- **BUG 2 fix live-verified** on the student's own `lo` run: 6 packets /
  3 requests / 3 replies, "Sent by me: 6, Received by me: 0".
- **Routing evidence**: `ip route get 127.0.0.1` → `local 127.0.0.1 dev lo
  table local` (route type `local`, no `via`, resolved from the `local` routing
  table which is consulted before `main`); `ip route get 8.8.8.8` →
  `via 192.168.8.1 dev wlp0s20f3`. Student then produced the synthesis in his
  own words: kernel looks at destination → routing table → "this is me" → never
  hands the packet to any interface but `lo`. Gap filled by mentor: captures are
  per-interface taps, so zero-on-Wi-Fi and six-on-lo are the same routing fact
  observed from two positions.
- **Sent/received semantics discussed and DECIDED**: `Sent 6 / Received 0` is a
  truthful report from a model whose assumption (source ≠ destination) loopback
  violates. Alternatives (6/6, 3/3) are choices, not corrections — 3/3 would
  wrongly conflate ICMP type with direction. **Decision: counters stay as-is;
  the limitation is documented knowledge, not a bug. Closed — do not reopen.**
- **Curiosity answered**: predictable network interface naming. `wlp0s20f3` =
  wireless, PCI bus 0, slot 20 (hex 0x14), function 3 — matches `lspci`
  `00:14.3 Intel Wi-Fi 6 AX201` on this machine; `enp0s31f6` matches `00:1f.6`.
  Stable-topology names replaced race-prone discovery-order names (`eth0`).
- **Pacing feedback from the student** (2:30am): too many check questions, too
  slow, drift into tool-semantics debate. Standing anti-drift rules added to
  `AGENTS.md` ("Pacing"). Lesson ended immediately on request.
