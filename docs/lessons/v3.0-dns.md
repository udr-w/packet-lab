## Session 1 narrative (2026-07-04, same day as v2.0 close)

### Warm-up: the order of everything

Question: with no ARP entry and no cached DNS answer, what crosses the radio
before `ping google.com`'s first ICMP packet, and in what order?

Student placed DNS before ARP and proposed broadcasting to find the DNS
server. The broadcast idea traced back to a real ambiguity in the question:
"no DNS answer cached" leaned on the **configuration vs cache** distinction,
which had never been taught. Student called this out; the defect was the
question's, not the student's.

- **Directive added to AGENTS.md ("Question quality")**: questions must state
  their givens completely, rely only on already-taught distinctions, and when
  a wrong answer traces to an ambiguous question, the question owns the
  fault.

The distinction, then the corrected picture:

- **Configuration** = "who do I ask" — the DNS server address, delivered by
  DHCP at join, lives as long as the lease. Contains no answers.
- **Cache** = "answers already given" — name→IP results with TTLs.
- Correct order: **ARP → DNS → ICMP**. Nothing can leave before ARP (the DNS
  query is itself a UDP packet needing a frame with the router's MAC); the
  ICMP packet cannot exist before DNS supplies its destination IP. Each step
  manufactures the address the next step's envelope requires. One ARP
  exchange, done first, serves everything after it.

### Theory covered

- Resolution chain: application → stub resolver → router (DHCP-configured)
  → upstream/authoritative.
- DNS rides **UDP port 53**: one tiny question, one tiny answer; a TCP
  handshake would triple the cost. Lost replies are simply re-asked.
- Record types: **A** (name → IPv4, forward) and **PTR** (IP → name,
  reverse — the query behind the viewer's device-name labels).
- Query IDs pair responses to questions (same trick as ICMP id/seq).
- Caching happens at every level of the chain.

### Observation: the resolver config surprise

Student ran `cat /etc/resolv.conf` and `resolvectl status`:

- `nameserver 127.0.0.53` — a **loopback** address (v1.2 knowledge applied:
  this traffic never touches hardware). A DNS server runs *inside* the
  laptop: **systemd-resolved**, the stub resolver ("resolv.conf mode: stub").
- The stub answers from cache when it can; on a miss it forwards per-link:
  `Link 3 (wlp0s20f3): Current DNS Server: 192.168.8.1` — the router, from
  DHCP. Other links (docker0, virbr0, tailscale0, veths) have no DNS scope —
  later roadmap chapters.
- Chain on this machine:
  `app → 127.0.0.53 via lo (cache hit = radio silent) → 192.168.8.1 via
  wlp0s20f3 (real UDP packet: ARP/frame and all) → upstream`.

### Prediction (answered correctly; capture pending)

Given `dig` twice for a never-seen name while capturing UDP 53 on
wlp0s20f3: student predicted first dig forces the question onto the radio
(cache empty), second is served from the stub's cache and the radio stays
silent. **Correct.** The live capture proving it opens the next session.
