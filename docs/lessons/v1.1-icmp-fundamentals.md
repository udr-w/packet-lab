Investigated with:

```bash

```

Conclusions:

- NAT network for KVM/QEMU virtual machines

### 3. First packet capture

Installed `tcpdump`, `wireshark`, `tshark`.

Observed ping packets directly with:

```bash
sudo tcpdump -i wlp0s20f3 icmp
```

### 4. First Python packet viewer

Created a single Python script which:

- launches tcpdump
- captures stdout
- parses ICMP packets
- prints a formatted table

Sample output shape:

| Time | Source | Destination | Protocol | Message |
|------|---------|-------------|----------|---------|
| 23:19:40.160 | 🌐 Udara-s-S24-Ultra | 🖥️ Me | ICMP | Echo Request |
| 23:19:40.160 | 🖥️ Me | 🌐 Udara-s-S24-Ultra | ICMP | Echo Reply |

The script resolved hostnames via reverse DNS, which already correctly resolved
`Udara-s-S24-Ultra`. Custom name mappings were therefore treated as optional.

### 5. Phone communication verified

Verified communication between the laptop and a Samsung S24 Ultra (running
Termux) using:

```bash
ping <laptop-ip>
```

This confirmed the parser worked on real, student-generated traffic.

### 6. Viewer refactor into modules + real ICMP verification

The original single-file viewer was split into focused modules:

```text
packetlab/
    capture.py   -> starts tcpdump and yields raw output lines
    parser.py    -> parses tcpdump text into IcmpPacket objects
    resolver.py  -> identifies local IPs and does reverse DNS lookups
    stats.py     -> counts ICMP packets and echo request/reply totals
    ui.py        -> renders Rich tables and statistics panels

scripts/
    packetlab.py -> command-line entry point
```

**Architectural decision:** `IcmpPacket` keeps the original raw tcpdump line in
`packet.raw`; parsed fields are added *beside* the raw data instead of replacing
it. This supports the learning goal — the student can compare our interpretation
against the real Linux capture output.

Viewer behavior at end of milestone:

- Must be started with `sudo`, then runs `tcpdump -i <interface> -nn -tttt -l icmp`.
- If started without root, prints the exact `sudo python3 ...` command instead of
  opening an empty live UI.
- Parses ICMP lines and displays a Rich live table.
- Shows counters: total packets, sent by me, received by me, echo requests, echo
  replies, and request/reply gap.
- Shows unparsed tcpdump lines so parser mismatches are visible rather than
  silently producing an empty table.
- Does not hide `tcpdump` error output, so wrong interface names / tcpdump errors
  are visible.

**Debugging notes captured this milestone:**

- Running `sudo tcpdump` inside `subprocess.Popen()` can fail silently or
  confusingly when sudo needs a password prompt. Cleaner educational design:
  require the whole script to run with privileges and keep `tcpdump` as a normal
  child process.
- Running Python with `sudo` can use a different package environment than the
  normal user. The student hit `ModuleNotFoundError: No module named 'rich'`
  under `sudo` even though normal `python3` could import Rich. `scripts/packetlab.py`
  now catches this and recommends `sudo apt install python3-rich`.

**Static verification run:**

```bash
python3 -m compileall packetlab scripts
```

Parser/statistics behavior tested against two sample tcpdump lines:

```text
Echo Request
Echo Reply
PacketStats(total=2, sent=1, received=1, echo_requests=1, echo_replies=1, other_icmp=0)
```

### Real capture verification (reported by the student)

- `sudo python3 scripts/packetlab.py wlp0s20f3` works after installing `python3-rich`.
- `ping -4 google.com` from the laptop showed:
  - `Me -> cm-in-f100.1e100.net` as Echo Request
  - `cm-in-f100.1e100.net -> Me` as Echo Reply
- Phone-to-laptop ping showed:
  - `Udara-s-S24-Ultra -> Me` as Echo Request
  - `Me -> Udara-s-S24-Ultra` as Echo Reply
- Router ping matched the student's prediction exactly:
  - 3 Echo Requests sent by the laptop
  - 3 Echo Replies received by the laptop
  - 6 total packets
  - request/reply gap 0
- Example combined totals observed: 18 packets, 9 sent by me, 9 received by me,
  9 requests, 9 replies, request/reply gap 0.

### UI cleanup after verification

- Stats panel now always renders the same stat lines, even when counts are zero.
- Rich `Live` now uses `screen=True` to reduce repeated partial panels in the
  scrollback while the display updates. (Note: this decision later caused the
  hidden-stderr bug investigated in v1.2 — see `docs/lessons/v1.2-loopback.md`.)

## Milestone outcomes

**What the student demonstrated (confirmed this milestone):**

- Correctly predicted request/reply counts for `ping -c 3 -4 google.com`.
- Correctly predicted request/reply counts for pinging the router.
- Explained phone-to-laptop ping direction: phone sends Echo Request, laptop
  sends Echo Reply.
- Understands ICMP uses IP directly and does not require TCP.
- Understands a remote IP destination (Google) differs from the local next hop
  (the router).

**Architectural decisions made:**

- Keep raw tcpdump lines beside parsed fields.
- Require the viewer itself to run with `sudo` instead of launching
  `sudo tcpdump` inside a subprocess. (Later revised in v1.2 to use `setcap`.)
- Keep dependency errors visible, especially `python3-rich` under `sudo`.

**Student preference recorded this milestone:** the student pushed back when the
lesson drifted into repository/module discussion; future sessions should stay
focused on Linux networking unless code details directly clarify packet behavior.

**Handoffs into later milestones (open at close of v1.1):**

- Do not spend lesson time on code architecture unless needed to explain Linux
  networking.
- Do not repeat confirmed basics (e.g. friendly names being derived from IPs).
- If a ping uses IPv6, the current IPv4 parser shows `IP6 ... ICMP6 ...` as an
  unparsed line. ICMPv6 should be taught later, not mixed into the IPv4 ICMP
  lesson.
- A better hostname resolver with optional `devices.conf` remains future work.
